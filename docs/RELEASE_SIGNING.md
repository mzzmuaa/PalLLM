# PalLLM Release Signing

Last audited: `2026-05-07`

This doc explains how PalLLM releases are verified by consumers and
how to sign one yourself. It covers the **digest layer** (always
emitted by the build) and the **signature layer** (opt-in per
maintainer). Both are independent — a release is valid with digests
only, and gains a stronger trust anchor when signed.

## Why this exists

PalLLM is an unaffiliated third-party project. Players install it by
downloading a zip from GitHub Releases, extracting it, and running
`play.bat`. Any step of that pipeline that can be tampered with
silently (MITM on the download, a compromised mirror, a malicious
fork of the repo) is a safety concern for the player. Digests +
signatures close that gap without requiring any runtime infrastructure.

## What ships with every release

The release archive always has an accompanying digest manifest:

- `SHA256SUMS` — canonical one-line-per-file SHA-256 digest set
- `SHA512SUMS` — same in SHA-512 (stronger hash, slightly larger)
- `checksums.json` — structured manifest with size bytes + digest +
  capture timestamp (for tooling that prefers JSON)

Generated by `scripts/compute-release-checksums.ps1` over
`artifacts/packaging/`. The same script now also writes
`Runtime/ReleaseEvidence/latest-artifact-integrity.json`, so
`GET /api/release/readiness` can report whether the digest manifests exist,
how many artifacts they covered, and whether detached signature files were
present beside them.

### Verifying a download (consumer)

```powershell
# Windows — PowerShell
Get-FileHash -Algorithm SHA256 .\PalLLM-v1.0.0.zip
# Compare the Hash field against SHA256SUMS.
```

```bash
# Linux / macOS
sha256sum -c SHA256SUMS
# Expected: PalLLM-v1.0.0.zip: OK
```

If the hash doesn't match, **stop**. Either the download is corrupt
or the file has been tampered with.

## Signing a release (maintainer)

Signing is an extra layer above digests. PalLLM recommends
`minisign` because it's small, fast, offline, and has a much simpler
key-management story than GPG. Both are supported — releases may
carry either or both.

### minisign (recommended)

```bash
# Once, per maintainer — generate your signing key pair
minisign -G -p palllm-release.pub -s palllm-release.key

# Share the public key (palllm-release.pub) via the project README
# + NOTICE.md and GitHub. NEVER publish the secret key.

# Each release, after compute-release-checksums.ps1:
minisign -S -s palllm-release.key -m artifacts/packaging/SHA256SUMS
# Produces artifacts/packaging/SHA256SUMS.minisig

# Consumers verify with:
minisign -V -p palllm-release.pub -m SHA256SUMS
# (minisig must be in the same directory)
```

### GPG (alternative)

```bash
# Once — generate a key, upload the public key to keys.openpgp.org
gpg --full-generate-key
gpg --armor --export you@example.com > palllm-release.asc

# Each release:
gpg --detach-sign --armor artifacts/packaging/SHA256SUMS
# Produces SHA256SUMS.asc

# Consumers verify with:
gpg --verify SHA256SUMS.asc SHA256SUMS
```

### Code-signing the EXE

For Windows SmartScreen reputation, the packaged `PalLLM.Sidecar.exe`
should be code-signed. This is a separate concern from the release-
archive signing above.

```powershell
# Using a code-signing certificate (EV or OV)
signtool sign /fd SHA256 /f mycert.pfx /p password /tr http://timestamp.digicert.com /td SHA256 `
    artifacts/packaging/PalLLM-v1.0.0/sidecar/publish/PalLLM.Sidecar.exe
```

A code-signed EXE still benefits from a signed `SHA256SUMS` — the
SmartScreen-signing layer doesn't replace the integrity check over
the full archive.

## PalLLM's signing policy (reference install)

- Digests are **always** published with every release.
- Tagged GitHub releases attest the digest-listed artifacts with
  the SHA-pinned `actions/attest` workflow action and GitHub OIDC/Sigstore
  provenance. Consumers can verify
  the downloaded zip with `gh attestation verify PalLLM-<tag>.zip --owner
  <owner>` after checking it against `SHA256SUMS`.
- Signatures are **best-effort**: depends on the maintainer who cuts
  the release having keys set up. Absence of a signature does not
  invalidate a release, but its presence lets verifiers close more
  attack surface.
- The canonical public key(s) are published in `SECURITY.md` and
  pinned in GitHub Releases descriptions.
- Keys expire after 24 months. Key rotations are announced in
  `CHANGELOG.md` under a dedicated "Key rotation" heading.

## What this does NOT do

- **No runtime verification at install time.** `play.bat` does not
  currently verify the digest of its own zip — the user is expected
  to have verified before unzip. A future pass can add a
  verify-install helper that re-hashes the extracted tree against a
  shipped manifest.
- **No transport-layer signature.** HTTPS + GitHub's own TLS already
  protect the in-flight download. The PalLLM-level signature protects
  against the case where an attacker controls the published artifact
  itself.
- **No package-manager integration.** PalLLM is not published on
  Chocolatey / winget / scoop / apt today. If that changes, the
  package definitions should pin the SHA-256 from `SHA256SUMS` so
  the package-manager path stays integrity-checked.

## Rotating a compromised key

1. Immediately revoke the key at its host (keys.openpgp.org for GPG).
2. Open a `CHANGELOG.md` entry under "Key rotation" with the
   revoked key fingerprint + the replacement key fingerprint.
3. Re-sign the currently-shipped release with the new key.
4. Update `SECURITY.md` with the new fingerprint.

## Verification quick-reference

| Goal | Command |
|---|---|
| Verify one file | `sha256sum -c SHA256SUMS` |
| Verify signed digest | `minisign -V -p palllm-release.pub -m SHA256SUMS` |
| List signed releases | `gh release list` (GitHub CLI) |
| Inspect embedded manifest | `unzip -p PalLLM-v1.0.0.zip RELEASE_PACKAGE_MANIFEST.json` |
