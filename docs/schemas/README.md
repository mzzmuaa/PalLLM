# JSON Schemas

Last audited: `2026-05-23`

Machine-readable contracts for PalLLM's wire-level data shapes.
Useful for:

- **Harvesters** lifting one piece into their own runtime
- **External tool authors** writing producers or consumers of the
  bridge envelope format
- **Validators** in CI pipelines that need to confirm a fixture
  conforms before a test run
- **AI agents** generating sample envelopes from a precise contract

The HTTP request / response shapes have their own machine-readable
contract: `docs/openapi/palllm-sidecar-v1.json` (regenerated by
`scripts/export-openapi.ps1`). The schemas here cover wire-level
shapes that don't sit on the HTTP surface — file-on-disk envelopes
and pack formats that the bridge and packs subsystems consume.

## Versioning

Schemas use the `$id` URL pattern
`https://palllm.dev/schemas/<name>-<vN>.schema.json`. The major
version (`v1`) is bumped only when an existing field is removed or
its type changes incompatibly. New optional fields are added in
place without a major bump (the `$schema` declaration uses
`https://json-schema.org/draft/2020-12/schema` so consumers
default to "ignore unknown properties").

## Verifying a payload

Any JSON Schema 2020-12 validator works. PowerShell + `Test-Json`
on .NET 7+:

```powershell
$schema = Get-Content docs/schemas/outbox-envelope.schema.json -Raw
Test-Json -Json (Get-Content "$env:LOCALAPPDATA\Pal\Saved\PalLLM\Bridge\Outbox\<envelope>.json" -Raw) -Schema $schema
```

Or with `ajv-cli`:

```bash
npx ajv-cli validate -s docs/schemas/outbox-envelope.schema.json -d ./envelope.json
```

PowerShell 7.4+ can also validate the read-only proof-status projection:

```powershell
pwsh ./scripts/pal-proof.ps1 -Json |
    Test-Json -SchemaFile docs/schemas/native-proof-status-v1.schema.json
```

## Contributing a new schema

1. Add the `<name>.schema.json` file under `docs/schemas/`.
2. Use `$id`: `https://palllm.dev/schemas/<name>-v1.schema.json`.
3. Use `$schema`: `https://json-schema.org/draft/2020-12/schema`.
4. Add a row to the index above naming the source-of-truth code file.
5. The `Drift_Path_references` audit gate confirms the code-file
   reference resolves; the `Drift_Dangling_markdown_links` gate
   confirms the schema link resolves.

## Why schemas live in `docs/`, not `schemas/` at root

Two reasons:

- **Co-located with the rest of the docs.** A harvester reading
  `docs/INDEX.md` finds the schemas by browsing.
- **Avoids a top-level directory dedicated to one consumer
  category.** PalLLM has tests/, scripts/, src/, mod/, docs/ at
  root — adding `schemas/` would add a sixth top-level concept,
  which raises the "where does X live?" cognitive cost.
